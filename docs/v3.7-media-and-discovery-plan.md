## 1. PDF books (paste-link flow, same as epub/txt)

User obtains PDFs themselves and pastes Drive/public links — same ingestion as
today, extended to `.pdf`.

- `lib/books.ts`:
  - `sniffFormat`: add `%PDF-` magic → `"pdf"`.
  - New `pdfText(bytes): Promise<string>`: lazy `await import("pdfjs-dist/legacy/build/pdf.mjs")`
    (same lazy-import pattern as jsdom/fflate — keeps cold routes fast).
    `getDocument({ data: bytes, disableWorker: true })`, loop pages,
    `page.getTextContent()`, join `item.str` with spaces, pages with `\n\n`.
    No text layer (scanned PDF) → throw "this PDF has no readable text layer"
    (lands as extraction failed — the designed link-only fallback).
    Then `chunkPlainText(text)`.
  - `loadBook`: drive + remote providers handle `"pdf"` like epub (title from
    filename; PDFs carry no reliable title metadata — skip metadata read).
  - `retryBookProbe`: accept pdf (probe succeeds if text extracts; store
    `format: "pdf"` on bookSource).
  - `DRIVE_CAP` 25MB stays (rejects huge PDFs with the existing message).
- `lib/types.ts`: `BookSource.format: "txt" | "epub" | "pdf"`.
- `app/api/library/route.ts`: remote-URL branch accepts `.pdf` extension.
- New dependency: `npm i pdfjs-dist` (pin latest v4.x; legacy build works in
  Node/serverless without canvas). If Vercel build complains about optional
  `canvas`, add it to `serverExternalPackages` in `next.config.ts`.
- Verify: a small public PDF (e.g. an arxiv.org PDF link) pastes → probe ok →
  reads page by page; a scanned/image PDF → "couldn't open" + Retry (expected).

## 3. Article discovery: real sources, grouped

`/api/discover?kind=article` today = s.jina.ai SERP (JINA_API_KEY) →
DuckDuckGo HTML fallback; results feel thin and Medium rarely surfaces.
Make it multi-source (all keyless unless noted), each `Promise.allSettled`,
merged, never erroring (empty results on total failure — existing contract):

- **Medium**: run the existing jina SERP a second time with
  `site:medium.com <q>` (DDG fallback likewise supports `site:`). Tag those
  results `siteName: "Medium"`. Ingestion already rewrites Medium → Freedium.
- **dev.to** (tech topics): real JSON API, no key —
  `https://dev.to/api/articles?per_page=10&tag=<best-guess tag>`; fallback
  `?top=7` query via their `?state=rising`. Map to results with
  `siteName: "dev.to"`, snippet = description.
- **Wikipedia** (evergreen background): REST API, no key —
  `https://en.wikipedia.org/w/rest.php/v1/search/page?q=<q>&limit=5`
  (set a descriptive UA). `siteName: "Wikipedia"`.
- Keep the general web SERP (jina/ddg) as the "Web" group.
- Response: results keep flat shape with `siteName` set;
  `components/SourceSearch.tsx` groups rows under group headers
  (Medium / dev.to / Wikipedia / Web) when kind=article. Videos tab unchanged
  apart from §2.
- Verify: article search for the user's real topic shows all groups; with
  JINA_API_KEY unset, DDG+dev.to+Wikipedia still produce results.

## 5. Embed non-YouTube videos

Today only YouTube embeds (kind "youtube"); other video URLs become articles.
Add an embed resolver and treat embeddable video hosts as video items (reuse
kind "youtube" to avoid a schema change — the check constraint allows only
article/youtube/book; siteName already shows the host).

- New `lib/embed.ts`: `videoEmbed(u: URL): { host: string; embedUrl: string } | null`
  - YouTube (existing logic delegates here), Vimeo `vimeo.com/<id>` →
    `https://player.vimeo.com/video/<id>`, Dailymotion
    `dailymotion.com/video/<id>` → `https://www.dailymotion.com/embed/video/<id>`,
    direct `.mp4/.webm/.ogv` URL → `<video controls>` (embedUrl = the URL).
  - User mentioned a source that sounded like "jitterbit" plus X (Twitter)
    and tool websites — **ask them for one real example link**, inspect its
    embed/oEmbed offering, and add that host's pattern here too.
- `app/api/library/route.ts` ingestion: if `videoEmbed()` matches → kind
  "youtube", siteName = host, thumbnail where cheaply available (YouTube
  oEmbed path stays; Vimeo/Dailymotion oEmbed is one fetch — optional).
  Transcript extraction stays YouTube-only; other hosts keep
  hasContent=false → discuss-from-title fallback (already designed).
- `components/ReaderView.tsx`: replace the YouTube-only embed with the
  resolver (iframe for vimeo/dailymotion, `<video controls>` for direct
  files).
- Verify: paste a Vimeo + a Dailymotion + a direct .mp4 link → card kind
  video, plays embedded; discussion works from title.
